# Impact History Guide

## Назначение

`impact` — это история изменений проекта: что делали, зачем делали, где меняли и когда это происходило.

`impact` не заменяет stable documentation. Guide-документы в `Final_Test_Hybrid/Docs/*` остаются каноническим описанием текущего поведения системы. `impact` нужен для трассировки решений, follow-up задач, расследований и повторного входа в контекст.

## Каноническое расположение

Все impact-файлы живут в `Final_Test_Hybrid/Docs/impact/`.

```text
Final_Test_Hybrid/Docs/impact/
├── ImpactHistoryGuide.md
├── impact-template.md
├── runtime/
│   └── archive/
├── execution/
│   └── archive/
├── diagnostics/
│   └── archive/
├── ui/
│   └── archive/
└── cross-cutting/
    └── archive/
```

Папки `archive/` используются только для исходных файлов после lossless rollup. Активные impact держать в корне соответствующего контура.

## Контуры

Использовать один из пяти контуров:

- `runtime` — reconnect/reset/error-flow, PLC gating, OPC runtime, execution safety.
- `execution` — orchestration, шаги, completion-flow, retry/skip, результаты, pipeline execution.
- `diagnostics` — Modbus, диагностические шаги, scanner/boiler lock, сервисные сценарии диагностики.
- `ui` — визуальные контуры, таблицы, вкладки, блокировки интерфейса, UX-поведение.
- `cross-cutting` — изменения, которые затрагивают несколько контуров сразу или меняют общий workflow/процесс.

Не дублировать один и тот же impact в нескольких контурах. Если работа реально многоконтурная, использовать только `cross-cutting` и перечислять затронутые контуры внутри файла.

## Что читать перед новым планом

Перед созданием нового плана выполнять один и тот же порядок:

1. Прочитать релевантные stable docs по затронутому контуру.
2. Прочитать последние релевантные active impact в том же контуре.
3. Прочитать последний релевантный `cross-cutting`, если задача затрагивает общий workflow, несколько контуров или продолжает старый общий change-set.
4. Если есть active rollup по той же теме, считать его основным входом в историю; `archive/` по умолчанию не читать.
5. В `archive/` идти только если текущая задача продолжает ту же цепочку изменений и active rollup не покрывает нужные детали.

Если по нужному контуру impact-файлов ещё нет, это нужно явно отметить в рабочем контексте как отсутствие history, а не пропускать планирование.

## Когда создавать новый impact

Новый impact создавать только после завершённого плана и завершённых проверок, если были реальные изменения в repo-tracked файлах или stable docs.

Не создавать impact для:

- чистого исследования без правок;
- разового code review без изменений;
- планирования без реализации;
- локальных временных команд, не изменивших repo-tracked состояние.

## Быстрый стоп-чек: когда impact точно не нужен

Если на любой вопрос ответ `нет`, новый отдельный impact не создавать:

- Есть ли реальные изменения в repo-tracked файлах?
- Изменились ли stable docs или другая проектная история, которую нужно сохранить?
- Завершился ли отдельный план / change-set, а не только исследование, вопрос, review или временная локальная проверка?
- Это новая отдельная цепочка изменений, а не продолжение того же active workstream?

Если работа продолжается в рамках того же active workstream и текущий impact всё ещё описывает ту же цепочку, обновлять существующий active impact вместо создания нового файла.

Правило хранения:

- один выполненный план = один отдельный impact-файл;
- имя файла: `YYYY-MM-DD-<slug>.md`;
- `slug` держать коротким и читаемым, в `kebab-case`;
- если задача многоконтурная, файл создавать только в `cross-cutting/`.

## Что обязательно должно быть в impact

Использовать `impact-template.md` как базовый шаблон. В каждом impact должны быть:

- `Контекст`;
- `Почему делали`;
- `Что изменили`;
- `Где изменили`;
- `Когда делали`;
- `Хронология`;
- `Проверки`;
- `Риски`;
- `Открытые хвосты`;
- `Связанные планы и документы`.

Для `rollup` обязателен дополнительный раздел `Сводит impact` с перечислением всех объединённых файлов.

В `Когда делали` и `Хронология` фиксировать major milestones, а не каждый shell-вызов:

- исследование;
- принятие решения;
- правки;
- проверки;
- финализация.

## Правила сжатия и объединения

Перед созданием нового impact проверить, нет ли safe кандидатов на сжатие в том же контуре.

Разрешён только один механизм: `lossless rollup + archive`.

### Когда rollup допустим

- impact относятся к одному контуру;
- impact относятся к одной теме, одному subsystem, одному плану, одному инциденту или явному follow-up той же цепочки;
- цепочка уже завершена и по ней не идёт активная работа;
- rollup не скрывает исходные даты, причины изменений, проверки и ссылки на документы.

### Как делать rollup

1. Создать новый файл `YYYY-MM-DD-rollup-<slug>.md` в active-папке контура.
2. Перенести в него общую историю, хронологию, ключевые изменения, проверки и остаточные риски.
3. Заполнить раздел `Сводит impact` списком всех исходных файлов.
4. Переместить исходные файлы в `Final_Test_Hybrid/Docs/impact/<контур>/archive/`.
5. Убедиться, что active rollup теперь достаточно для чтения перед новым планом.

### Что запрещено

- объединять active impact по незавершённой работе;
- сводить unrelated задачи только ради уменьшения числа файлов;
- объединять разные контуры вне заранее общего `cross-cutting` workstream;
- удалять исходные impact без следа;
- терять ссылки на планы, инциденты, проверки или stable docs.

## Минимальный чек перед закрытием задачи

- stable docs обновлены, если изменился канонический контур поведения;
- новый impact создан, если change-set реально изменил repo-tracked состояние;
- impact лежит в правильном контуре;
- для многоконтурной задачи использован `cross-cutting`;
- candidates на safe rollup проверены;
- если rollup делался, исходники доступны в `archive/`.
